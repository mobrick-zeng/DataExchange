## 一、一句話總覽

跨銀行「前置調解債權申報」平台，**已從前端 Demo 演進為前後端整合的全棧系統**：
最大債權行建立案件並代填各行債權 → 其他債權行確認／回報異議 → 進入「還款中」逐月更新 → 結清或終止。

**與舊文件最大差異：** 舊 `README.md` / `04-current-status.md` 描述的「純前端、資料存 localStorage、不接後端／資料庫、Email OTP 自助註冊」**已不再成立**。前端的 localStorage Mock 資料層已**全數移除**，改為呼叫後端 REST API。

## 二、技術架構（現況）

| 層 | 技術 | 位置 |
|---|---|---|
| 前端 | React 18 + TypeScript + Vite 5 + Tailwind 3 + React Router 6 + Zustand | 專案根目錄 `src/` |
| 前端資料存取 | 統一經 `src/services/api.ts`（`apiFetch`，附 JWT）呼叫後端；**已無任何 localStorage 資料層** | `src/services/` |

| 部署 | 本機開發（Vite + npm）或全棧 Docker（db + api + web + Caddy 自動 HTTPS） | `docker-compose.fullstack.yml` |

### 認證與帳號模型（邀請制）
- **登入**：銀行 + Email + 密碼三層驗證 → 回傳 JWT（8 小時）。
- **建立帳號**：管理員發一次性邀請碼（效期 3 天、免寄信）→ 受邀者於 `/register?code=…` 設定姓名與密碼完成註冊，帳號即 `ACTIVE`。
- **密碼重設**：使用者於「忘記密碼」送出申請 → 管理員核發一次性重置碼 → 使用者以 `/register?code=…`（用途辨識為重置）僅設定新密碼。
- **已移除**：舊版公開自助註冊、Email 6 碼 OTP 驗證頁、Demo 信箱、待審核頁、舊版重設密碼頁。

### 案件狀態機（後端模型）
```
DRAFT → PENDING_CONFIRMATION → IN_REPAYMENT → SETTLED / TERMINATED
```
> 舊前端的 `DRAFT → PENDING_DECLARATION → IN_PROGRESS → COMPLETED → CLOSED`（自行申報／退回補件）**已淘汰**，對應的 `DeclarationPage` 已移除。

## 四、如何執行

### 方式一：本機開發（本次採用，開發最快）

需 Node.js 20+ 與 Docker Desktop。

```bash
# 1) 啟動資料庫（Docker，Postgres 對外埠 5544）
cd backend
docker compose up -d

# 2) 套用 migration 與種子（首次或資料重置時）
npx prisma migrate deploy
npm run prisma:seed          # SEED_MODE 預設 demo，建立 5 組示範帳號

# 3) 啟動後端 API（http://localhost:4000）
npm run dev

# 4) 啟動前端（另開終端，於專案根目錄）
cd ..
npm run dev                  # http://localhost:5173
```

- 後端連線設定見 `backend/.env`（`DATABASE_URL` 指向 localhost:5544、`CORS_ORIGIN` 已含 5173）。
- 前端 API 位址由 `VITE_API_BASE` 決定，未設定時預設 `http://localhost:4000`。

### 方式二：全棧 Docker（一鍵，含自動 HTTPS）

```bash
# Demo 版（含 5 組示範帳號）
docker compose -f docker-compose.fullstack.yml -p mediation-demo up -d --build
# 瀏覽器開 https://localhost（本機自簽憑證，點「繼續前往」）

# 正式版（只建 admin + 銀行主檔，其餘帳號一律邀請制）
SEED_MODE=production docker compose -f docker-compose.fullstack.yml -p mediation-prod up -d --build
```
上線前設定（`JWT_SECRET`／`DB_PASSWORD`／網域憑證／備份）請複製根目錄 `.env.production.example` 為 `.env` 並填入正式值。

## 五、開發與部署工作流程（dev ↔ fullstack）

### 核心觀念：一份原始碼、兩種跑法
dev 與 fullstack **讀的是同一份原始碼**（`src/`、`backend/src/`、`backend/prisma/`），差別只在「怎麼跑」：

- **dev 模式**：直接執行原始碼（Vite / tsx），存檔即熱重載，連本機 Docker 的 Postgres。適合**頻繁修改、除錯**。
- **fullstack 模式**：用 `--build` 把同一份原始碼**編譯打包成 Docker 映像**（前端 → Nginx 靜態、後端 → Node），由 Caddy 提供 HTTPS。適合**驗收、展示、部署**。

因此 dev 的修改**不需要「合併」進 fullstack**——它們本來就是同一份檔案。下次對 fullstack 下 `up --build`，你的最新修改就會自動編進映像。

### 建議的開發循環
```bash
# 1) 平常開發（快速迭代）
cd backend && docker compose up -d     # 起 dev 資料庫（Postgres :5544）
cd backend && npm run dev              # 後端 API :4000（tsx watch 熱重載）
npm run dev                            # （另開終端、於根目錄）前端 :5173（Vite HMR）
#   → 在 http://localhost:5173 邊改邊看

# 2) 若改到資料表結構（backend/prisma/schema.prisma）
cd backend && npm run prisma:migrate   # 產生遷移檔到 backend/prisma/migrations/（務必保留）

# 3) 要驗收整套 / 部署時（以最新原始碼重建映像）
docker compose -f docker-compose.fullstack.yml -p mediation-demo up -d --build
#   → 在 https://localhost 驗證打包後的成品
```

> **`--build` 就是 dev → fullstack 唯一的「合併」動作**（以目前原始碼重新編譯映像）。
> 資料表結構變更則透過**遷移檔**傳遞：fullstack 後端映像啟動時會自動 `prisma migrate deploy`，套用 `backend/prisma/migrations/` 內的遷移。

### ⚠️ 不建議「直接在 fullstack 狀態下開發」
fullstack 映像是編譯／壓縮後的 production 版，無法直接編輯；每次改動都要 `up --build` 全量重建（慢、無熱重載），進容器改檔也會在重建時消失。**開發請用 dev 模式，fullstack 只拿來驗收／部署。**

### 兩個必知差異（同一份碼、不同環境）
1. **資料庫是分開的兩套。** dev 用 volume `backend_pgdata`（:5544）；fullstack 有自己的 `db` 服務與各自的 volume（依 `-p` 專案名，如 `mediation-demo_pgdata`）。**dev 灌的測試資料不會出現在 fullstack**；fullstack 啟動時自行跑 seed（demo 建 5 組示範帳號、production 只建 admin＋銀行）。此為正常隔離，非資料遺失。
2. **前端 API 位址用 build 參數切換，非改程式碼。** dev：`VITE_API_BASE=http://localhost:4000`（預設）；fullstack：build 時帶 `VITE_API_BASE=""` → 走同源 `/api`，由 Caddy 代理到後端。同一份程式碼兩邊皆可跑。
