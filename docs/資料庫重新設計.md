# 資料庫重新設計（v2）

> 更新：**2026-07-22**。本文件為 v2 資料模型的設計藍本與決策紀錄。
> 對應 schema：[`backend/prisma/schema.prisma`](../backend/prisma/schema.prisma)、種子：[`backend/prisma/seed.ts`](../backend/prisma/seed.ts)。
> 基本業務流程不變，僅重新設計資料內容；**原有資料完全清除（greenfield，無資料遷移）**。

---

## 一、設計目標與重大變更

| 主題 | v1（舊） | v2（新） |
|---|---|---|
| 債務人辨識 | 存 `debtorName` / `debtorId`（個資） | **不存個資**，以 `(courtCode, docNumber)` 法院公文文號辨識 |
| 帳號所屬銀行 | `appliedBankCode` + `approvedBankCode` 雙欄位 | 單一 `bankCode` |
| 帳號啟用（正式區） | 立即 ACTIVE | **啟用碼首次登入 → 強制設密碼 → ACTIVE**（碼由外部管道遞送，免 SMTP） |
| 個資同意 | 無 | 啟用時**強制勾選**，記 `consentedAt` + `consentVersion` |
| 銀行/法院 | 僅 3 家佔位 | **完整參考表 + `isActive`**，後台可啟用管理 |
| 債權容器 | `CreditorDeclaration` 獨立表 | 併入 `CaseParticipantBank`（1:1） |
| 金額加總 | 落地存 `externalTotal`/`totalAmount` | 編輯期即算、**確認當下凍結** `confirmedClaimAmount` |
| 還款 | 每期手填 `outstandingBalance` | 主辦輸入實收總額 → 依 `planRatio` **自動攤提**；餘額推導 |
| 稽核 | `AuditLog` + `AccountApprovalLog` 兩表 | **單一 append-only `AuditLog`** |
| OTP/邀請 | `EmailOtp` + `Invitation` | 合併為 `AccessCode`（啟用碼／重置碼） |

## 二、實體關係總覽（12 張表）

```
Court（法院主檔, isActive）───< Case
Bank（金融機構主檔, isActive）──< User
                              └─< Case(mainBank) / CaseParticipantBank
User ─< AccessCode / PasswordResetRequest / Notification / AuditLog / (createdCases…)
Case ─< CaseParticipantBank ─< CreditItem
     │                       └─< RepaymentAllocation
     └─< RepaymentPeriod ─────< RepaymentAllocation
```

- **參考主檔**：`Bank`、`Court`（皆 `isActive` 控制可用）
- **帳號**：`User`、`AccessCode`、`PasswordResetRequest`
- **案件**：`Case`、`CaseParticipantBank`（含債權容器）、`CreditItem`
- **還款**：`RepaymentPeriod`（每案每期實收）、`RepaymentAllocation`（攤提到各行）
- **通知/稽核**：`Notification`、`AuditLog`

金額 `Decimal(21,4)`；比例 `planRatio` `Decimal(9,6)`（各行加總＝1，應用層驗）。

## 三、四大流程

### 1. 帳號啟用
- **Demo**：種子帳號直接 `ACTIVE`（密碼 `Demo@1234`）。
- **正式區**：admin 建帳號 → `PENDING_ACTIVATION` + 產生 `AccessCode(ACTIVATION)`（存雜湊、有效期）→ 明碼經**外部管道**交付 → 使用者於登入頁「選銀行 + email + 啟用碼當密碼」→ 導向啟用頁**設新密碼 + 勾選個資同意** → `ACTIVE`（記 `activatedAt`/`consentedAt`/`consentVersion`，碼標記 `usedAt`）。碼一次性、逾期失效、錯誤計入登入鎖定。

### 2. 最大債權行（主辦）
建立案件（`DRAFT`，選法院＋填文號）→ 代填各行債權明細（`CreditItem`）→ 設定還款計畫（`monthlyInstallment`/期數/起始月）與各行 `planRatio`（手動、加總＝1）→ 邀請其他債權行 → **發布**（`PENDING_CONFIRMATION`）。發布後 `planRatio` 鎖定（僅 ADMIN 可改），主辦僅能修正被異議的資料。

### 3. 其他債權行
收通知 → 檢視自家債權明細 → **確認**（凍結該行 `confirmedClaimAmount`）或**回報異議**（`DISPUTED` + 理由，待主辦修正後重新確認）。全數確認 → 案件轉 `IN_REPAYMENT`、寫 `case.confirmedAt` 與 `totalDebtAmount`。

### 4. 逐月還款
主辦每期只輸入 `period` + `actualReceivedTotal`（+備註）→ 系統依 `planRatio` 產生每行 `RepaymentAllocation`（`plannedAmount` = 月應還×比例、`actualAmount` = 實收×比例）；**尾差記入主辦** `roundingAdjustment`，有尾差時 `hasRoundingAdjust=true`（前端提醒）。畫面對照表（計算得出、不落地）：
- 剩餘債務 = `confirmedClaimAmount` − 累計 `actualAmount`
- **計畫完成率** = Σ實際 / Σ原定應還；**債權回收率** = Σ實際 / `confirmedClaimAmount`
主辦另可 `SETTLED` 結清或 `TERMINATED` 毀諾終止。

## 四、Demo 全階段示範資料（`seed.ts`）

`SEED_MODE=demo`（預設）建立：
- **參考主檔**（兩模式共用）：35 家金融機構（僅 `012/808/812`＋`PLATFORM` 啟用）、23 所法院（僅 `TPD/SLD` 啟用）；代碼為真實值（玉山＝808、永豐＝807 停用）。
- **帳號**（密碼 `Demo@1234`）：admin、稽核、012/808/812 承辦，各為 `ACTIVE`；另有 1 個 `PENDING_ACTIVATION` 帳號 `pending.co@bank.local` + 啟用碼 `ACT-DEMO-808`（示範啟用流程，啟動時印於 console）。
- **案件（涵蓋每個階段）**：DRAFT、PENDING_CONFIRMATION（部分確認）、PENDING_CONFIRMATION（含異議）、IN_REPAYMENT（2 期、含短繳尾差）、SETTLED、TERMINATED。

`SEED_MODE=production`：僅建立主檔 + 1 組 admin，其餘帳號以啟用碼建立。

> 案件種子有防重（`case.count()>0` 即略過），容器每次重啟執行 seed 不會重複。

## 五、清庫重建（greenfield）

因資料完全清除，採全新 migration 基線。首次建立：

```bash
# 開發機（Docker DB 已啟動）
cd backend
docker compose up -d
npx prisma migrate dev --name v2_init   # 產生全新基線 migration
npm run prisma:seed
```

全棧/OCI 部署：容器啟動時 `prisma migrate deploy` 會套用該基線，再 `prisma db seed`。
重置為正式區流程見 [`系統現況與執行說明.md`](./系統現況與執行說明.md) 第三節。
